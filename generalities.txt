
GENERAL RECAP:
-> Per ogni step, avere un solo target e una sola azione (non gestire la simultaneità in modo asimmetrico)
-> Tipo di input:
	1. Immagine: va male 
	2. Immagine + grafo: un po' meglio, ma l'immagine non serviva a nulla 
	3. Solo grafo: 
	   3.1 Grafo con relazioni spaziali (in_front_of, ecc): ancora sbaglia
	   3.2 Grafo con sides accessibili specificati (box_2 sides:-left some spec): molto meglio 


CURRENT PROBLEMS:
-> Detection pose + dimensions 
-> Analyze scenarios in which objects are graspable in different ways, like both with the single-arm and with both arms (look at notes -> sphere and box cases)
-> How to create a good graph for occlusions 
-> How to handle simultaneous actions -> the model has proven to be not able to handle multiple # from __future__ import annotations
-> Place camera on the end-effector


-> Decide how to handle different modules


ARCHITECTURE UP TO NOW: 
SAM (?) + SCENE PERCEPTION + PLANNING + SIMULATENOUS + DEPLOY + VALIDATOR 

where:
- SAM (?) detects and lables the objects present in the scene 
- SCENE PERCEPTION handles: - detection of objects poses and dimensions; -creation of occlusion graph 
- PLANNING outputs action primitives and action_features (please consider the possibility of having a separate module for the action_features)
- SIMULATENOUS matches actions that can be executed in parallel 
- DEPLOY 
- VALIDATOR to check action success or failure

- handover (? - ma necessita della divisione della zona in tre parti)
